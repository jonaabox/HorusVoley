## Feature #4 — Reemplazar `alert()`/`confirm()` con modales

### Problema
El sistema usa `alert()` y `confirm()` nativos del navegador en 4 lugares. Son bloqueantes, no personalizables y rompen la experiencia visual de la app.

### Solución
Un hook `useConfirm` + componente `ConfirmModal` que replica la API de `confirm()` de forma asíncrona.

### Archivos nuevos
- `src/hooks/useConfirm.js` — hook que gestiona el estado del modal. Devuelve `{ confirm, ConfirmModal }`. La función `confirm(options)` es async y retorna `true/false`.
- `src/components/ConfirmModal.jsx` — modal con header oscuro (`#1e3a5f`) consistente con los modales existentes. Props: `title`, `message`, `confirmLabel`, `variant` (`danger` | `info`), `onConfirm`, `onCancel`.
  - `danger`: botón de confirmación rojo, ícono ⚠️ — para acciones destructivas
  - `info`: botón de confirmación primario, ícono ℹ️, sin botón cancelar — para alertas informativas

### Archivos modificados
| Archivo | Cambio |
|---|---|
| `src/pages/Alumnos.jsx:166` | `confirm()` → `await confirm({ title: 'Eliminar alumno', message: '...', variant: 'danger' })` |
| `src/pages/Pagos.jsx:145` | `confirm()` → `await confirm({ title: 'Eliminar pago', ... , variant: 'danger' })` |
| `src/pages/Campanas.jsx:333` | `confirm()` → `await confirm({ title: 'Eliminar grupo', ..., variant: 'danger' })` |
| `src/pages/Dashboard.jsx:146` | `alert()` → `await confirm({ title: 'Atención', message: '...', variant: 'info' })` |

### Uso en cada página
```js
const { confirm, ConfirmModal } = useConfirm()

// En el handler:
const ok = await confirm({ title: '...', message: '...', variant: 'danger' })
if (!ok) return

// Al final del JSX:
<ConfirmModal />
```

## Feature #1 — Batch notificación deudores

### Problema
No hay manera de saber rápidamente qué alumnos están por vencer sin entrar a cada perfil. Se necesita una alerta proactiva para facilitar el cobro.

### Solución
Un contexto global `DebtAlertContext` que calcula al cargar la app qué alumnos activos no pagaron el mes actual y tienen vencimiento próximo. Muestra una notificación en el sidebar con link directo a Campañas con la lista pre-cargada.

### Lógica del cálculo
Un alumno entra en la lista de notificación si:
- `estado = 'activo'`
- No tiene pago registrado para el mes/año actual
- Los días hasta el vencimiento son ≤ `dias_aviso_vencimiento` (configurable) o ya venció

Reutiliza la función `calcularMesesDeuda` ya existente en el proyecto.

### Modelo de datos
Nueva clave en tabla `configuracion`:
```sql
INSERT INTO configuracion (clave, valor) VALUES ('dias_aviso_vencimiento', '5');
```

### Archivos nuevos
- `src/context/DebtAlertContext.jsx` — provider que consulta Supabase al montar (alumnos activos + pagos del mes + config), calcula la lista de deudores próximos y la expone via context. Se refresca al navegar a la sección.

### Archivos modificados
| Archivo | Cambio |
|---|---|
| `src/App.jsx` | Envuelve la app con `<DebtAlertProvider>` |
| `src/components/Sidebar.jsx` | Badge amarillo con contador en ítem Campañas + mini-banner colapsable con resumen y botón "Ver lista →". Al hacer click navega a `/campanas` con `state: { deudores: [...] }` via `useNavigate` |
| `src/pages/Campanas.jsx` | Lee `location.state.deudores` con `useLocation`. Si existe, pre-carga la pestaña de Alumnos con esos contactos y un mensaje template de recordatorio de cuota |
| `src/pages/Configuracion.jsx` | Agrega campo editable para `dias_aviso_vencimiento` junto al día de vencimiento |
| `supabase/migrations/` | Nueva migration con el INSERT de `dias_aviso_vencimiento` |

## Feature #3 — Horarios y Grupos A/B

### Problema
No existe forma de asignar alumnos a un horario específico. Los sábados hay dos turnos (14:30–16:30 y 16:30–18:30) y se necesita poder diferenciarlos en la lista de alumnos y al tomar asistencia.

### Solución
Nueva tabla `horarios` con los dos turnos como datos iniciales. Columna `horario_id` en `alumnos`. Selector en el formulario de alumno. Columna visible en la tabla. Filtro por grupo en Asistencia.

### Modelo de datos

**Tabla nueva: `horarios`**
```sql
CREATE TABLE horarios (
  id          uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  nombre      text NOT NULL,        -- "Grupo A", "Grupo B"
  dia_semana  text NOT NULL,        -- "Sábado"
  hora_inicio time NOT NULL,        -- 14:30 / 16:30
  hora_fin    time NOT NULL,        -- 16:30 / 18:30
  created_at  timestamptz DEFAULT now()
);
```

Seed inicial:
```sql
INSERT INTO horarios (nombre, dia_semana, hora_inicio, hora_fin) VALUES
  ('Grupo A', 'Sábado', '14:30', '16:30'),
  ('Grupo B', 'Sábado', '16:30', '18:30');
```

**Tabla modificada: `alumnos`**
```sql
ALTER TABLE alumnos ADD COLUMN horario_id uuid REFERENCES horarios(id) ON DELETE SET NULL;
```
Nullable: alumnos existentes y nuevos sin asignar quedan con `NULL` (se muestra "Sin asignar").

### UI

**Lista de alumnos:** nueva columna "Grupo" con badge de color (azul para A, violeta para B, gris para sin asignar). Muestra nombre del grupo + hora de inicio (ej. `A · 14:30`).

**Formulario de alumno:** selector de horario con opciones cargadas desde Supabase: Grupo A / Grupo B / Sin asignar.

**Asistencia:** botones de filtro "Todos / Grupo A · 14:30 / Grupo B · 16:30" que filtran la lista de alumnos mostrada.

### Archivos involucrados
| Archivo | Cambio |
|---|---|
| `supabase/migrations/` | Tabla `horarios` + ALTER TABLE alumnos + seed + RLS |
| `src/pages/Alumnos.jsx` | Carga horarios desde Supabase, agrega selector al form, agrega columna Grupo a la tabla |
| `src/pages/Asistencia.jsx` | Carga horarios, agrega filtro de grupo, filtra lista según selección |

## Feature #5 — Biblioteca de materiales (PDF)

### Problema
No existe forma de compartir material de estudio con los alumnos. Se necesita una biblioteca permanente de PDFs que permita envío masivo por WhatsApp.

### Solución
Nueva página `Materiales` con biblioteca de PDFs almacenados en Supabase Storage. Cada PDF tiene un link público. El botón "Enviar" abre un modal que permite elegir destinatarios y genera el mensaje de WhatsApp con el link, reutilizando la infraestructura de Campañas.

### Modelo de datos

**Supabase Storage: bucket `materiales`**
- Acceso público (para generar URLs compartibles por WhatsApp sin autenticación del receptor)
- RLS: solo usuarios autenticados pueden subir/eliminar

**Tabla nueva: `materiales`**
```sql
CREATE TABLE materiales (
  id           uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  nombre       text NOT NULL,
  descripcion  text,
  storage_path text NOT NULL,  -- path en el bucket
  public_url   text NOT NULL,  -- URL pública del archivo
  created_at   timestamptz DEFAULT now()
);
```

### Página Materiales (`src/pages/Materiales.jsx`)

**Lista:** tarjetas con nombre, descripción, fecha de subida. Acciones: Enviar | Eliminar (con `ConfirmModal` de Feature #4).

**Upload:** botón "+ Subir PDF" abre modal con input de nombre, descripción opcional y file picker (solo `.pdf`). Al guardar: sube a Storage → obtiene URL pública → inserta en tabla `materiales`.

**Envío:** botón "Enviar" en cada material abre un modal de destinatarios con tres opciones de selección:
- Todos los alumnos activos
- Por grupo (Grupo A / Grupo B)
- Selección manual (checkboxes)

Al confirmar, genera el mensaje:
```
Hola {{nombre_completo}}, te compartimos el siguiente material de estudio:
[nombre del PDF]
{{url}}
```
Y abre el `EnvioModal` de Campañas con la lista de contactos y el mensaje pre-armado.

### Extracción de `EnvioModal`
El componente `EnvioModal` definido dentro de `Campanas.jsx` se extrae a `src/components/EnvioModal.jsx` para ser reutilizado en Materiales.

### Archivos involucrados
| Archivo | Cambio |
|---|---|

| `src/pages/Materiales.jsx` | Nueva página completa |
| `src/components/EnvioModal.jsx` | Extraído de Campanas, ahora componente independiente |
